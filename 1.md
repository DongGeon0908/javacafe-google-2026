### 우리 조직은 프로그래밍과 소프트웨어 엔지니어링 중에 어떤 곳에 포커싱을 두는가?

---

## 🧭 0. 이 챕터의 핵심 메시지

이 책의 1장은 단순히 “코드를 잘 짜는 법”을 말하지 않는다.  
대신 다음 질문을 던진다:

> “우리는 지금 ‘프로그래밍’을 하고 있는가,  
> 아니면 ‘소프트웨어 엔지니어링’을 하고 있는가?”

그리고 그 차이를 **시간, 규모, 트레이드오프**라는 관점에서 설명한다.

---

## 1. 프로그래밍 vs 소프트웨어 엔지니어링

### ✔ 프로그래밍

- 특정 문제를 해결하기 위해 코드를 작성하는 행위
- 주 관심사:
  - 기능 구현
  - 로직 완성
  - “지금 돌아가는가?”

👉 즉, **단기적인 문제 해결 중심**

---

### ✔ 소프트웨어 엔지니어링

- 소프트웨어를 **지속 가능하게 유지하기 위한 모든 활동**
- 포함 범위:
  - 코드
  - 설계
  - 테스트
  - 배포
  - 운영
  - 팀/조직 구조

👉 즉, **시간이 지나도 유지되는 시스템을 만드는 것**

---

### ✔ 핵심 차이

| 구분 | 프로그래밍 | 소프트웨어 엔지니어링 |
| ---- | ---------- | --------------------- |
| 관점 | 현재       | 시간 전체             |
| 목표 | 동작       | 지속 가능성           |
| 범위 | 코드       | 조직 + 시스템         |

---

## 2. 소프트웨어 엔지니어링의 3가지 축

책에서는 소프트웨어 엔지니어링을 다음 3가지로 정의한다:

### ✔ 1) 시간 (Time)

- 소프트웨어는 반드시 변경된다
- 오늘의 최적이 내일의 최악이 될 수 있다

👉 중요한 질문  

> “이 코드는 1년 뒤에도 유지될 수 있는가?”

---

### ✔ 2) 규모 (Scale)

- 사용자 수 증가
- 코드베이스 증가
- 개발자 수 증가

👉 문제

- 작은 시스템에서 잘 동작하던 구조가
- 큰 시스템에서는 깨진다

---

### ✔ 3) 트레이드오프 (Trade-off)

모든 선택에는 비용이 따른다

예:

- 성능 ↑ → 복잡도 ↑
- 개발 속도 ↑ → 유지보수성 ↓
- 안정성 ↑ → 개발 생산성 ↓

👉 핵심

> 정답은 없고, 상황에 맞는 균형이 중요

---

## 3. 지속 가능성 (Sustainability)

### ✔ 정의

> 소프트웨어의 기대 수명 동안 발생하는 모든 “가치 있는 변경”을 수용할 수 있는 능력

---

### ✔ 왜 중요한가?

현실에서는:

- 기술이 바뀜 (Java, Spring, Infra)
- 비즈니스 요구가 바뀜
- 조직이 바뀜

👉 결국

> “변경을 얼마나 잘 받아들이는가”가 핵심

---

### ✔ 경험 사례 (번장)

과거:

- Spring 2.x + Java 8
- 강한 커플링 + 의존성 문제

업그레이드 시:

- 의존성 충돌
- 대규모 수정 필요
- 매우 높은 비용 발생

---

### ✔ 이후 개선

- 의존성 관리 정책 도입
- 구조 개선
- 점진적 업그레이드 전략

👉 결과

- Boot 3 + Java 17 전환은 훨씬 수월

---

### ✔ 인사이트

> “고통을 없앨 수는 없다.  하지만 고통을 줄이는 구조는 만들 수 있다”

---

## 4. 하이럼의 법칙 (Hyrum’s Law)

### ✔ 정의

> API의 모든 관찰 가능한 동작은  
> 누군가에 의해 의존된다

---

### ✔ 의미

- 문서화되지 않은 동작도 사용된다
- 사용자가 많을수록 더 강하게 나타난다

---

### ✔ 실제 사례 (번장)

- v1 API 사용 없음 판단
- 제거 진행

👉 결과

- 예상하지 못한 곳에서 에러 발생
- 관계사 장애까지 영향

---

### ✔ 교훈

- “우리가 모르는 사용자”는 반드시 존재
- API 제거는 기술 문제가 아니라 **사회적 문제**

---

### ✔ 대응 전략

- Deprecation 기간 확보
- 로그 기반 실제 사용 추적
- 점진적 제거

---

## 5. 단기 vs 장기 관점

### ✔ 단기 (Programming 중심)

- 빠른 개발
- POC
- 비즈니스 검증

---

### ✔ 장기 (Engineering 중심)

- 금융 시스템 (10~30년)
- 높은 안정성 요구
- 지속적인 변경 대응

---

### ✔ 핵심 질문

> “이 코드는 언제까지 살아야 하는가?”

👉 이 질문에 따라 설계가 완전히 달라진다

---

## 6. 변경(Change)에 대한 관점

### ✔ 특징

- 변경은 필연적이다
- 하지만 비용이 매우 크다

---

### ✔ 잘못된 접근

- “변경이 없을 것이다”라고 가정

예:

- 보안 취약점 (Heartbleed 등)
- CPU 취약점 (Meltdown, Spectre)

👉 현실

> 세상은 계속 바뀐다

---

### ✔ 올바른 전략

- 변경을 줄이되
- 변경 가능성에는 대비

---

### ✔ 구조적 해결 방법

- 낮은 결합도 (Loose Coupling)
- 높은 응집도
- 모듈화
- 인터페이스 기반 설계

---

### ✔ 핵심 인사이트

> “변경을 막는 것이 아니라  
> 변경의 영향 범위를 줄이는 것이 핵심”

---

## 7. 확장성 (Scale)

### ✔ 확장의 의미

- 트래픽 증가
- 코드 증가
- 팀 증가

---

### ✔ 문제

- 사람이 늘어나면:
  - 커뮤니케이션 비용 증가
  - 코드 충돌 증가

---

### ✔ 해결 방법

- 플랫폼 팀 구성
- 공통 인프라 제공
- 개발 표준화

---

### ✔ 인사이트

> “확장은 기술 문제가 아니라 조직 문제다”

---

## 8. 비욘세 규칙 (Beyoncé Rule)

### ✔ 정의

> “If you liked it, then you should have put a test on it”

---

### ✔ 의미

- 중요한 코드라면 반드시 테스트 필요
- 테스트 없는 코드는 변경 비용 증가

---

### ✔ 현실적인 문제

- 레거시 코드에 테스트 추가 비용 큼

---

### ✔ 전략

- 신규 코드 → 반드시 테스트
- 기존 코드 → 점진적 적용

👉 예:

- Strangler Pattern
- 테스트 가능한 구조로 리팩토링

---

## 9. 자동화 / 일관성 / 전문성

### ✔ 자동화

- 빌드
- 테스트
- 배포

👉 사람 개입 최소화 → 안정성 증가

---

### ✔ 일관성

- 코드 스타일
- API 규칙
- 아키텍처

👉 유지보수 비용 감소

---

### ✔ 전문성

- 각 영역 전문가 필요
- 단, 사일로 방지 필요

---

## 10. 좋은 엔지니어링 의사결정

### ✔ 정의

> 다양한 근거를 바탕으로 균형점을 찾는 것

---

### ✔ 특징

- 정답 없음
- 상황 의존적
- 트레이드오프 고려 필수

---

### ✔ 핵심 문장 (발표용 추천)

> “좋은 엔지니어링은 정답을 찾는 것이 아니라  
> 균형점을 찾는 것이다”

---

## 11. 효율 vs 단순성

### ✔ 문제

- 최적화 → 복잡도 증가
- 단순성 → 성능 저하 가능

---

### ✔ 예시

- JOIN 제거
- 한방 쿼리 대신 여러 쿼리

---

### ✔ 인사이트

> “미래 유지보수 비용까지 포함해서 판단해야 한다”

---

## 12. Fork vs Import

### ✔ Fork

- 빠름
- 독립적

👉 단점:

- 유지보수 어려움

---

### ✔ Import

- 일관성 유지
- 업데이트 가능

---

### ✔ 선택 기준

| 상황      | 선택   |
| --------- | ------ |
| 단명 코드 | Fork   |
| 장수 코드 | Import |

---

## 13. 마무리 질문 (발표 마무리용)

- 이 시스템은 얼마나 오래 살아야 하는가?
- 우리는 변경 비용을 얼마나 고려하고 있는가?
- 지금의 선택이 미래에 어떤 영향을 주는가?
- 우리는 프로그래밍을 하고 있는가, 엔지니어링을 하고 있는가?
